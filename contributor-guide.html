00 개요

이 가이드를 읽기 전에 — 프로젝트가 어떤 구조로 돌아가는지, 그리고 리포트를 만들 때의 작업 흐름을 한 눈에 파악한다.

각 기여자가 자유 양식으로 작성한 리서치 자료를 AI를 활용해 HTML 리포트로 변환하고, 공통 디자인 시스템이 적용된 웹 아카이브에 게시한다. 리포트가 게시될 때마다 핵심 내용을 추린 인스타그램 카드뉴스를 별도로 제작해 아카이브로의 유입을 유도한다.

STEP 1 · 입력
자유 양식 자료 + 프롬프트 입력
본인이 평소 쓰는 AI(Claude, ChatGPT, Gemini 등 무관)에 원본 자료와 아래 03장의 고정 프롬프트를 함께 입력한다.
STEP 2 · 미리보기
_PREVIEW 버전으로 먼저 확인
AI가 실제 서비스 스타일이 적용된 미리보기 파일을 준다. 다운로드해서 아무 폴더에나 두고 더블클릭으로 열면 된다 (인터넷 연결 필요). 폴더 구조를 맞출 필요 없음.
STEP 3 · 수정
AI와 대화하며 편집 / 메모장으로 오탈자 수정
도식화·구조 변경은 AI에게 자연어로 요청. 오탈자·문장 다듬기 같은 단순 텍스트 수정은 메모장으로 직접 고쳐도 된다.
STEP 4 · 최종본 요청
"최종본 줘"라고 반드시 요청
수정이 끝났다고 미리보기 파일을 그대로 올리면 안 된다. AI에게 최종본을 요청하면 _PREVIEW 표시와 절대경로가 제거되고, 저장소에 맞는 상대경로로 바뀐 업로드용 파일이 나온다.
STEP 5 · 업로드
GitHub에 업로드
최종본 파일을 다운로드해 GitHub 저장소 루트에 업로드하면 자동으로 아카이브에 반영되고 Vercel을 통해 배포된다.
01 배포 구조

최종본 리포트 파일이 GitHub에 올라가는 순간부터 실제 웹사이트에 반영되기까지의 자동화 흐름을 설명한다.

Step 01
최종본 HTML 파일 업로드
기여자가 GitHub 웹 UI에서 저장소 루트에 reports_...html 파일을 업로드하고 main 브랜치에 커밋한다.
Step 02
GitHub Actions 자동 실행
reports_*.html 파일 변경이 감지되면 build.yml 워크플로우가 트리거된다.
Step 03
build.py 실행 — index.html 자동 갱신
build.py가 저장소 루트를 스캔해 reports_로 시작하는 파일들의 report-meta 블록에서 메타데이터를 추출하고, index.html의 REPORTS 배열을 자동으로 업데이트한 뒤 커밋한다.
Step 04
Vercel 자동 배포
main 브랜치에 새 커밋이 생기면 Vercel이 감지해 자동 배포한다. 업로드 후 약 1~2분 내 실제 사이트에 반영된다.
주의

GitHub 웹 UI 업로드만 사용한다. index.html은 직접 편집하지 않는다 — build.py가 자동으로 덮어쓴다.

덮어써도 안전합니다

기존 리포트를 수정해서 같은 파일명으로 재업로드해도 예전 내용이 사라지지 않는다. GitHub이 모든 이전 버전을 자동으로 보관하기 때문이다. 해당 파일 페이지 상단의 History 버튼을 누르면 지금까지의 모든 수정 이력과 예전 내용을 언제든 확인할 수 있다.

02 파일 구조

GitHub 저장소는 하위 폴더 없이 평탄화되어 있다. 파일명 접두어만으로 역할을 구분한다.

{프로젝트명}/ ← 폴더 구분 없이 전부 루트에 위치 ├── index.html ← 아카이브 메인 페이지 (build.py가 자동 갱신, 직접 수정 금지) ├── build.py ← 루트 스캔 → index.html REPORTS 배열 갱신 ├── .github/workflows/ │ └── build.yml ← GitHub Actions 워크플로우 ├── assets_report.css ← 리포트 공용 스타일시트 ├── assets_report.js ← 리포트 공용 스크립트 (topic/title 자동 주입) ├── reports_insights_..._1.html ├── reports_issues_..._1.html └── reports_sectors_..._1.html ← 개별 리포트, 계속 늘어남
왜 이런 구조인가

모든 파일이 같은 위치(저장소 루트)에 있어서, 리포트 파일을 로컬 어디에 두든 CSS/JS 참조가 항상 정상 작동한다. 파일명 접두어(assets_, reports_)만으로 역할을 구분한다.

파일명 규칙

리포트 파일명은 아래 요소를 언더스코어(_)로 연결한다. 슬러그 내 단어 구분은 하이픈(-)을 사용한다.

접두어
항상 reports_ 로 시작
cat
insights / issues / sectors 중 하나
slug
주제를 영어로 요약. 단어는 하이픈으로 연결. 시리즈 번호는 넣지 않는다 (예: kb-balhae-1이 아니라 kb-balhae)
seq
같은 슬러그 안에서 몇 번째 글인지 나타내는 순번(1, 2, 3...). 기여자가 직접 부여한다. 시리즈 파트든, 나중에 같은 주제로 새로 쓴 별개의 글이든 이 번호 하나로 구분한다. 2편 이상으로 이어지는 시리즈물에만 붙인다. 한 편으로 끝나는 단일 리포트는 seq를 생략한다 — 파일명 끝에 번호를 붙이지 않는다.

예시: reports_sectors_land-reclamation.html (단일 리포트, seq 없음), reports_insights_kb-balhae_2.html (시리즈 2번째 글, seq 있음)

수정 시 규칙

기존 리포트 내용을 고칠 때는 같은 파일명으로 덮어써서 재업로드한다. seq 번호를 새로 매기지 않는다. report-meta의 date도 최초 업로드일 그대로 유지하고 수정일로 바꾸지 않는다 — 아카이브 정렬이 이 값을 기준으로 하기 때문이다. 실제 수정 시점과 변경 이력은 GitHub History에서 항상 확인 가능하니 걱정하지 않아도 된다.

report-meta 블록

각 리포트 파일 최상단 <!DOCTYPE html> 바로 뒤에 작성한다. build.py가 이 블록을 읽어 index 페이지를 갱신한다. (이 블록의 필드 구성은 이전과 동일하다 — cat 값도 insights처럼 복수형 그대로 쓴다)

report-meta 형식
<script type="application/json" id="report-meta"> { "title": "리포트 제목 (한국어, 문장형)", "topic": "주제어 (짧게, 예: 공유수면매립사업)", "cat": "insights / issues / sectors 중 하나", "date": "YYMMDD", "author": "author ID", "published": true } </script>

카테고리 정의

Cat설명예시 주제
insights개념·구조 해설인프라펀드 밸류에이션, 리츠 구조
issues시장·정책·딜 이슈에셋라이트 전략, 월드컵 인프라 금융
sectors자산군·산업군 정리공유수면매립, 물류센터
03 리포트 제작 가이드

원본 자료를 AI로 변환해 미리보기로 확인하고, 수정한 뒤, 최종본을 업로드하는 절차를 단계별로 안내한다.

AI에 첨부할 파일

어떤 AI를 쓰든(Claude, ChatGPT, Gemini 등) 요청할 때 아래 3개 파일을 반드시 첨부한다. 본인 원본 자료(PDF, 마크다운 등)는 별도로 함께 첨부한다.

assets_report.css
디자인 시스템 CSS — 시각 스타일 참조용
assets_report.js
공용 스크립트 — 구조 참조용
레퍼런스 리포트 .html 1개
HTML 구조·말투·포맷 참조용. 기존 리포트 파일 중 아무거나 하나 사용한다.

AI에 입력할 프롬프트

아래 프롬프트를 복사해 대화창에 붙여넣는다. 노란색으로 표시된 부분만 본인 정보로 채워서 사용한다. 이 프롬프트는 미리보기 → 수정 → 최종본 요청까지 전체 흐름을 AI에게 미리 지시해두는 내용이다.

Prompt — 대화창에 붙여넣기
첨부한 파일은 다음과 같아: 1. assets_report.css — 디자인 시스템 CSS 2. assets_report.js — 공용 스크립트 3. [레퍼런스 리포트 파일명].html — 구조·포맷 참조용 기존 리포트 4. [원본 자료 파일명] — 이번에 리포트로 만들 원본 자료 원본 자료를 바탕으로 리포트 HTML 파일을 만들어줘. 아래 규칙과 작업 순서를 반드시 지켜줘. --- [파일명 규칙 — 최종본 기준] reports_{cat}_{slug}_{seq}.html (2편 이상 이어지는 시리즈물) reports_{cat}_{slug}.html (한 편으로 끝나는 단일 리포트 — seq 생략) - cat: insights / issues / sectors 중 하나 - slug: 주제를 영어로 요약, 단어는 하이픈(-)으로 연결. 시리즈 번호는 넣지 않음 - seq: 시리즈물일 때만 붙임. 같은 슬러그 안에서 몇 번째 글인지 (내가 직접 지정함): [seq 번호 — 예: 1]. 단일 리포트면 이 부분 자체를 생략하고 파일명을 slug 뒤에서 바로 끝낸다. [report-meta 블록] 파일 최상단 <!DOCTYPE html> 바로 뒤에 아래 형식으로 작성: <script type="application/json" id="report-meta"> { "title": "리포트 제목 (한국어, 문장형)", "topic": "주제어 (짧게, 예: 공유수면매립사업)", "cat": "insights / issues / sectors 중 하나", "date": "최초 작성일 YYMMDD (신규 작성이면 오늘 날짜, 기존 리포트 수정이면 처음 게시했던 날짜 그대로 유지 — 수정일로 바꾸지 않음)", "author": "[본인 author ID — 예: jbchoi, jhahn, sjahn]", "published": true } </script> [HTML 구조] 첨부한 레퍼런스 리포트의 HTML 구조를 그대로 따른다. topbar, article.article, footer.article-foot 구조를 유지한다. article-head 안에는 반드시 아래 두 줄을 비워둔다 (report.js가 자동 주입): <p class="a-topic"></p> <h1 class="a-title"></h1> a-meta에는 날짜만 표시하고, author(작성자)는 화면에 표시하지 않는다. 작성자 정보는 report-meta JSON의 author 필드에만 남긴다. [분량] 본문 텍스트 기준 공백 제외 2,500~5,000자. - 짧으면 살을 붙여달라고 요청한다. - 길면 자연스러운 지점에서 분할해 파일을 2개로 나눠달라고 요청한다. [말투] - "~이다", "~한다", "~있다" 형태의 건조한 단문 서술체 - "~요", "~습니다" 등 구어체 금지 - 감상·평가 표현 없이 사실과 구조 중심으로 서술 [구성] - 섹션 수, 목차 구조, callout·formula·table 사용 여부는 내용에 맞게 자유롭게 결정 - lead 문단(p.lead)으로 시작하는 것을 권장 --- [★ 작업 순서 — 반드시 이 순서를 따라줘] 1단계 — 미리보기(_PREVIEW) 버전 먼저 출력 - 파일명 끝에 "_PREVIEW"를 붙인다 (예: reports_sectors_xxx_1_PREVIEW.html) - CSS/JS 참조 경로를 아래 절대경로로 바꿔서 넣는다: <link rel="stylesheet" href="https://srcplus.vercel.app/assets_report.css"> <script src="https://srcplus.vercel.app/assets_report.js"></script> - 화면(또는 파일 상단)에 "PREVIEW — 업로드 금지" 라는 표시를 눈에 띄게 넣는다. - 이 미리보기 파일은 내가 다운로드해서 로컬에서 더블클릭으로 열어 확인할 것이다. 인터넷 연결만 있으면 폴더 위치 상관없이 실제 서비스와 동일하게 보인다. 2단계 — 자연어 수정 - 내가 "3번 섹션 표로 바꿔줘" 같은 방식으로 수정을 요청하면, 수정된 새 _PREVIEW 파일을 다시 준다. - 수정을 마칠 때마다 매번 응답 끝에 이렇게 리마인드해줘: "▶ 확인 후 만족스러우면, 업로드 전에 반드시 '최종본 줘'라고 요청하세요. 지금 이 _PREVIEW 파일은 업로드하면 안 됩니다." 3단계 — 최종본 요청 시 - 내가 "최종본 줘" 또는 이와 유사한 표현으로 요청하면: a. 파일명에서 "_PREVIEW"를 뗀다 → reports_{cat}_{slug}_{seq}.html (시리즈물) 또는 reports_{cat}_{slug}.html (단일 리포트) b. CSS/JS 경로를 절대경로에서 상대경로로 되돌린다: href="assets_report.css" src="assets_report.js" c. "PREVIEW — 업로드 금지" 표시를 제거한다. d. 완성된 최종 파일명을 명확히 알려준다. - 이 최종본 파일이 GitHub에 실제로 업로드할 파일이다.

미리보기 확인 방법

1단계에서 받은 _PREVIEW 파일은 다음과 같이 확인한다.

  1. 파일을 다운로드한다 (바탕화면, 다운로드 폴더 등 아무 곳이나 상관없음)
  2. 더블클릭해서 브라우저로 연다
  3. 인터넷에 연결되어 있으면 실제 서비스와 동일한 디자인으로 보인다
폴더 구조 신경 쓸 필요 없음

이 파일은 스타일을 실제 배포된 사이트에서 실시간으로 불러오기 때문에, zip을 다운받아 압축을 풀고 정확한 폴더에 넣는 과정이 필요 없다. 파일 하나만 다운로드해서 열면 된다.

이 파일은 업로드 금지

_PREVIEW가 붙은 파일, 또는 링크에 srcplus.vercel.app이 하드코딩된 파일은 절대 GitHub에 올리지 않는다. 실수로 올라간 걸 발견하면 내용을 고칠 필요 없이 그 파일만 삭제하면 된다 (파일명에 PREVIEW가 그대로 보이니 식별이 쉽다).

검토 및 수정

미리보기를 열어보고 마음에 들 때까지 아래 두 방식으로 반복 수정한다.

AI에게 재요청 — 도식화, 레이아웃, 문단 구조 수정

대화창에 자연어로 요청한다. AI가 수정된 새 _PREVIEW 파일을 다시 준다. 매 응답 끝에 "최종본을 요청하라"는 안내가 따라온다.

메모장으로 직접 편집 — 오탈자, 단순 텍스트 수정

_PREVIEW 파일을 메모장(Mac은 텍스트 편집기)으로 열어 Ctrl+F(Mac: Cmd+F)로 찾아 고쳐도 된다. 다만 이렇게 직접 고친 파일도 _PREVIEW 상태 그대로이므로, 수정이 끝나면 반드시 AI에게 다시 "최종본 줘"라고 요청해서 절대경로를 상대경로로 바꾼 버전을 받아야 한다.

가장 중요한 한 가지

수정을 아무리 많이 해도, 업로드 직전에는 반드시 "최종본 줘"라고 요청해서 받은 파일만 GitHub에 올린다. _PREVIEW가 붙어있거나 절대경로가 남아있는 파일을 그대로 올리면 안 된다.

완성 후 업로드

  1. GitHub 저장소 접속
  2. 우측 상단 Add file → Upload files
  3. 최종본 HTML 파일(파일명에 _PREVIEW 없는 것)을 저장소 루트에 드래그앤드롭
  4. 하단 커밋 메시지 입력란에 작업 유형에 맞게 작성 (아래 표 참고)
  5. Commit directly to main branch 선택 후 커밋
메시지사용 시점예시
add:신규 리포트 추가add: reports_issues_asset-lite_1.html
update:기존 리포트 수정 후 재업로드update: reports_issues_asset-lite_1.html
remove:리포트 삭제remove: reports_issues_asset-lite_1.html
주의

GitHub 웹 UI 업로드만 사용한다. 업로드 후 약 1~2분 내 자동 배포되며, GitHub Actions 탭에서 진행 상황을 확인할 수 있다.

04 카드뉴스 제작

완성된 리포트를 바탕으로 인스타그램용 카드뉴스를 제작하고 게시하는 절차를 안내한다.

구조 개요

카드뉴스는 표지 1장 + 본문 최대 5장으로 구성한다. 리포트의 모든 내용을 담으려 하지 않는다. 게시물 본문이나 댓글에 리포트 전체 링크를 달아 전체 내용은 아카이브에서 읽도록 유도한다.

슬라이드역할내용
표지 (1장)주목 유도리포트 제목 또는 핵심 질문
본문 (최대 5장)내용 전달각 장마다 하나의 포인트

콘텐츠 구성 — AI 활용

Prompt — 콘텐츠 구성 요청
첨부한 파일은 완성된 리포트 HTML이야. 이 리포트를 인스타그램 카드뉴스로 만들려고 해. 아래 조건에 맞게 슬라이드 구성안을 제안해줘. - 표지 1장 + 본문 최대 5장, 총 6장 이내 - 리포트 전체를 요약하는 게 아니라, 흥미롭거나 유익한 포인트만 골라서 - 각 슬라이드마다: 제목(한 줄), 본문(2~4줄 이내) - 본문은 수치·구조·핵심 개념 중심으로 간결하게 - 말투: "~이다", "~한다" 단문 서술체 슬라이드별로 [제목 / 본문] 형식으로 정리해줘.

템플릿 편집

AI가 제안한 구성안을 바탕으로 PowerPoint 템플릿(cardnews-template.pptx, Google Drive 보관)을 편집한다.

건드리지 말 것

검은색 그라데이션 박스, 좌상단 브랜드명(SRC Plus), 텍스트 폰트 유형 및 크기는 그대로 유지한다.

인스타그램 게시

  1. PowerPoint에서 파일 → 내보내기 → 이미지로 저장 (PNG 권장)
  2. 인스타그램 게시물 작성 시 슬라이드 이미지를 순서대로 첨부
  3. 게시물 본문 또는 댓글에 리포트 전체 링크를 추가
