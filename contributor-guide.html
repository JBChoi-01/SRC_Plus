00 개요

이 가이드를 읽기 전에 — 프로젝트가 어떤 구조로 돌아가는지 한 눈에 파악한다.

각 기여자가 자유 양식으로 작성한 리서치 자료를 AI를 활용해 HTML 리포트로 변환하고, 공통 디자인 시스템이 적용된 웹 아카이브에 게시한다. 리포트가 게시될 때마다 핵심 내용을 추린 인스타그램 카드뉴스를 별도로 제작해 아카이브로의 유입을 유도한다.

리서치
자료 작성
기여자가 관심 주제를 자유 양식으로 조사·정리한다. PDF, 마크다운, 텍스트 등 형식 무관.
변환
AI로 HTML 리포트 제작
원본 자료를 Claude에게 전달해 공통 디자인 시스템이 적용된 HTML 파일로 변환한다. 목차·도식·표 등 구성은 기여자가 자유롭게 결정한다.
게시
웹 아카이브 업로드
완성된 HTML 파일을 GitHub에 업로드하면 자동으로 아카이브 메인 페이지에 반영되고 Vercel을 통해 배포된다.
확산
인스타그램 카드뉴스
리포트 핵심 내용을 카드뉴스로 제작해 인스타그램에 게시한다. 게시물에 아카이브 링크를 달아 전체 리포트로 유입을 유도한다.
01 배포 구조

리포트 파일이 GitHub에 올라가는 순간부터 실제 웹사이트에 반영되기까지의 자동화 흐름을 설명한다.

Step 01
리포트 HTML 파일 업로드
기여자가 GitHub 웹 UI에서 reports/ 폴더에 파일을 업로드하고 main 브랜치에 커밋한다.
Step 02
GitHub Actions 자동 실행
reports/**.html 경로 변경이 감지되면 build.yml 워크플로우가 트리거된다.
Step 03
build.py 실행 — index.html 자동 갱신
build.py가 reports/ 폴더를 스캔해 각 파일의 report-meta 블록에서 메타데이터를 추출하고, index.html의 REPORTS 배열을 자동으로 업데이트한 뒤 커밋한다.
Step 04
Vercel 자동 배포
main 브랜치에 새 커밋이 생기면 Vercel이 감지해 자동 배포한다. 업로드 후 약 1~2분 내 실제 사이트에 반영된다. GitHub Actions 탭에서 진행 상황을 확인할 수 있다.
메타데이터 추출 규칙

각 리포트 파일 최상단의 report-meta JSON 블록이 소스다. build.py는 이 블록에서 title, cat, date, author, published 값을 읽어 index 페이지의 리스트를 구성한다. 블록이 없거나 JSON 파싱에 실패하면 해당 파일은 목록에서 제외된다.

주의

GitHub 웹 UI 업로드만 사용한다. 로컬 git push와 혼용하면 충돌이 발생한다. index.html은 직접 편집하지 않는다 — build.py가 자동으로 덮어쓴다.

02 폴더 구조

GitHub 리포지토리의 폴더 구조와 각 파일의 역할을 설명한다.

{프로젝트명}/ ├── index.html ← 아카이브 메인 페이지 (build.py가 자동 갱신) ├── build.py ← reports/ 스캔 → index.html REPORTS 배열 갱신 ├── .github/ │ └── workflows/ │ └── build.yml ← GitHub Actions 워크플로우 ├── assets/ │ ├── report.css ← 리포트 공용 스타일시트 │ └── report.js ← 리포트 공용 스크립트 (topic/title 주입 등) └── reports/ └── {cat}_{slug}_{YYMMDD}_{author}.html

파일명 규칙

리포트 파일명은 아래 4개 요소를 언더스코어(_)로 연결한다. 슬러그 내 단어 구분은 하이픈(-)을 사용한다.

cat
insights / issues / sectors 중 하나
slug
주제를 영어로 요약. 단어는 하이픈으로 연결. 예: land-reclamation
YYMMDD
작성 날짜 6자리. 예: 260624
author
작성자 ID. 예: jbchoi, jhahn, sjahn

예시: issues_asset-lite_260624_sjahn.html

report-meta 블록

각 리포트 파일 최상단 <!DOCTYPE html> 바로 뒤에 작성한다. build.py가 이 블록을 읽어 index 페이지를 갱신한다.

report-meta 형식
<script type="application/json" id="report-meta"> { "title": "리포트 제목 (한국어, 문장형)", "topic": "주제어 (짧게, 예: 공유수면매립사업)", "cat": "insights / issues / sectors 중 하나", "date": "YYMMDD", "author": "author ID", "published": true } </script>

카테고리 정의

Cat설명예시 주제
insights 개념·구조 해설 인프라펀드 밸류에이션, 리츠 구조
issues 시장·정책·딜 이슈 에셋라이트 전략, 월드컵 인프라 금융
sectors 자산군·산업군 정리 공유수면매립, 물류센터
03 리포트 제작 가이드

원본 자료를 Claude를 활용해 HTML 리포트로 변환하고 업로드하는 절차를 단계별로 안내한다.

Claude에 첨부할 파일

Claude에게 요청할 때 아래 3개 파일을 반드시 첨부한다. 본인 원본 자료(PDF, 마크다운 등)는 별도로 함께 첨부한다.

report.css
디자인 시스템 CSS — 시각 스타일 참조용
report.js
공용 스크립트 — 구조 참조용
레퍼런스 리포트 .html 1개
HTML 구조·말투·포맷 참조용. 기존 리포트 파일 중 아무거나 하나 사용한다.

Claude에 입력할 프롬프트

아래 프롬프트를 복사해 Claude 대화창에 붙여넣는다. 노란색으로 표시된 부분만 본인 정보로 채워서 사용한다.

Prompt — Claude에 붙여넣기
첨부한 파일은 다음과 같아: 1. report.css — 디자인 시스템 CSS 2. report.js — 공용 스크립트 3. [레퍼런스 리포트 파일명].html — 구조·포맷 참조용 기존 리포트 4. [원본 자료 파일명] — 이번에 리포트로 만들 원본 자료 원본 자료를 바탕으로 리포트 HTML 파일을 만들어줘. 아래 규칙을 반드시 지켜줘. --- [파일 규칙] - 파일명: {cat}_{slug}_{YYMMDD}_{author}.html - cat: insights / issues / sectors 중 하나 - slug: 주제를 영어로 요약, 단어는 하이픈(-)으로 연결 - YYMMDD: 오늘 날짜 (예: 260624) - author: [본인 author ID — 예: jbchoi, jhahn, sjahn] [report-meta 블록] 파일 최상단 <!DOCTYPE html> 바로 뒤에 아래 형식으로 작성: <script type="application/json" id="report-meta"> { "title": "리포트 제목 (한국어, 문장형)", "topic": "주제어 (짧게, 예: 공유수면매립사업)", "cat": "insights / issues / sectors 중 하나", "date": "YYMMDD", "author": "author ID", "published": true } </script> [CSS·JS 경로] - <link rel="stylesheet" href="../assets/report.css"> - <script src="../assets/report.js"></script> [HTML 구조] 첨부한 레퍼런스 리포트의 HTML 구조를 그대로 따른다. topbar, article.article, footer.article-foot 구조를 유지한다. article-head 안에는 반드시 아래 두 줄을 비워둔다 (report.js가 자동 주입): <p class="a-topic"></p> <h1 class="a-title"></h1> [분량] 본문 텍스트 기준 공백 제외 2,500~5,000자. - 완성 후 이 범위보다 짧으면 살을 붙여달라고 요청한다. - 이 범위보다 길면 자연스러운 지점에서 분할해 파일을 2개로 나눠달라고 요청한다. [말투] - "~이다", "~한다", "~있다" 형태의 건조한 단문 서술체 - "~요", "~습니다" 등 구어체 금지 - 감상·평가 표현 없이 사실과 구조 중심으로 서술 [구성] - 섹션 수, 목차 구조, callout·formula·table 사용 여부는 내용에 맞게 자유롭게 결정 - lead 문단(p.lead)으로 시작하는 것을 권장 --- 완성된 파일의 파일명도 함께 알려줘.

검토 및 수정

Claude가 HTML 파일을 만들어주면 바로 업로드하지 말고 내용을 먼저 검토한다. 수정 방식은 두 가지로 나뉜다.

직접 편집 — 텍스트 수정, 오타 교정

메모장 또는 VS Code로 HTML 파일을 열어 직접 수정한다. 오탈자, 날짜·수치·고유명사 오류, 문장 다듬기 등 텍스트 레벨 변경이 해당한다. Ctrl+F(Mac: Cmd+F)로 수정할 텍스트를 찾아 고친다.

Claude에게 재요청 — 도식화, 레이아웃 수정

현재 대화창에 수정할 HTML 파일을 첨부하고 자연어로 요청한다. Claude가 수정된 새 버전의 HTML 파일을 반환하면 기존 파일을 덮어쓴다.

요청 예시

"3번 섹션 내용을 표(table)로 정리해줘"

"2번과 3번 섹션 사이에 callout 박스 추가해줘, 내용은 ~"

"lead 문단을 좀 더 구체적인 사례로 시작하도록 수정해줘"

주의

수정 요청 시 HTML 파일을 반드시 다시 첨부한다. Claude는 이전 대화의 파일을 기억하지 못한다.

로컬 미리보기

HTML 파일을 단독으로 열면 CSS가 로드되지 않아 텍스트만 깨져 보인다. 이상한 게 아니다. 정상적으로 보려면 GitHub에서 리포지토리 전체를 zip으로 다운받아 압축 해제한 뒤, 그 안의 reports/ 폴더에 파일을 넣고 열면 된다. assets/와 reports/ 폴더가 같은 레벨에 있으면 어디서든 정상 렌더링된다.

완성 후 업로드

  1. GitHub 리포지토리 접속
  2. reports/ 폴더 클릭
  3. 우측 상단 Add file → Upload files
  4. 완성된 HTML 파일 드래그앤드롭
  5. 하단 커밋 메시지 입력란에 작업 유형에 맞게 작성 (아래 표 참고)
  6. Commit directly to main branch 선택 후 커밋

커밋 메시지는 GitHub이 강제하는 형식이 아니라 팀원끼리 히스토리를 알아보기 위한 표지다. 날짜·작성자·변경 파일 목록은 GitHub이 자동으로 기록한다.

메시지사용 시점예시
add: 신규 리포트 추가 add: issues_asset-lite_260624_sjahn.html
update: 기존 리포트 수정 후 재업로드 update: issues_asset-lite_260624_sjahn.html
remove: 리포트 삭제 remove: issues_asset-lite_260624_sjahn.html
주의

GitHub 웹 UI 업로드만 사용한다. 로컬 git push와 혼용하면 충돌이 발생한다. 업로드 후 약 1~2분 내 자동 배포되며, GitHub Actions 탭에서 진행 상황을 확인할 수 있다.

04 카드뉴스 제작

완성된 리포트를 바탕으로 인스타그램용 카드뉴스를 제작하고 게시하는 절차를 안내한다. 카드뉴스는 리포트 전체 내용을 담는 것이 아니라, 흥미롭거나 유익한 내용을 간추려 독자를 리포트로 유입시키는 역할을 한다.

구조 개요

카드뉴스는 표지 1장 + 본문 최대 5장으로 구성한다. 리포트의 모든 내용을 담으려 하지 않는다. 읽는 사람이 "더 알고 싶다"고 느낄 만한 핵심 포인트만 추린다. 게시물 본문이나 댓글에 리포트 전체 링크를 달아 전체 내용은 아카이브에서 읽도록 유도한다.

슬라이드역할내용
표지 (1장) 주목 유도 리포트 제목 또는 핵심 질문. 독자가 스크롤을 멈추게 만드는 한 문장
본문 (최대 5장) 내용 전달 각 장마다 하나의 포인트. 수치·구조·핵심 개념 중심으로 간결하게

콘텐츠 구성 — AI 활용

Claude 대화창에 완성된 리포트 HTML 파일을 첨부하고 아래와 같이 요청한다. AI가 카드뉴스에 담을 내용과 슬라이드별 구성안을 제안해준다.

Prompt — 콘텐츠 구성 요청
첨부한 파일은 완성된 리포트 HTML이야. 이 리포트를 인스타그램 카드뉴스로 만들려고 해. 아래 조건에 맞게 슬라이드 구성안을 제안해줘. - 표지 1장 + 본문 최대 5장, 총 6장 이내 - 리포트 전체를 요약하는 게 아니라, 흥미롭거나 유익한 포인트만 골라서 - 각 슬라이드마다: 제목(한 줄), 본문(2~4줄 이내) - 본문은 수치·구조·핵심 개념 중심으로 간결하게 - 말투: "~이다", "~한다" 단문 서술체 슬라이드별로 [제목 / 본문] 형식으로 정리해줘.

템플릿 편집

AI가 제안한 구성안을 바탕으로 PowerPoint 템플릿(cardnews-template.pptx)을 편집한다.

수정할 것

우측 상단 카테고리 텍스트

표지 슬라이드: 제목 텍스트

본문 슬라이드: 제목 텍스트, 본문 텍스트

각 슬라이드 배경 이미지 (아래 이미지 교체 방법 참고)

건드리지 말 것

검은색 그라데이션 박스 — 위치, 크기, 투명도 모두 유지

좌상단 브랜드명(SRC Plus) — 폰트, 크기, 위치 모두 유지

텍스트 폰트 유형 및 크기 — 내용만 교체, 서식은 건드리지 않는다

배경 이미지 교체

각 슬라이드의 배경 이미지는 슬라이드 주제와 어울리는 3:4 비율 이미지로 교체한다. AI 이미지 생성 도구(Midjourney, ChatGPT, Adobe Firefly 등 자유롭게 선택)를 활용하거나 무료 스톡 사이트(Unsplash, Pexels 등)에서 찾아도 된다.

  1. 슬라이드 주제에 맞는 이미지를 준비한다 (비율: 3:4 세로형)
  2. PowerPoint에서 해당 슬라이드의 기존 배경 이미지를 클릭해 선택
  3. 선택된 이미지를 삭제하고, 새 이미지를 삽입(삽입 → 그림)
  4. 이미지 크기를 슬라이드 전체에 맞게 조정
  5. 이미지를 우클릭 → 맨 뒤로 보내기 선택
  6. 그라데이션 박스와 텍스트가 이미지 위에 올바르게 표시되는지 확인
이미지 프롬프트 예시 (AI 생성 시)

"aerial view of offshore wind farm at sunset, vertical 3:4 ratio, cinematic, realistic photography"

AI 생성 이미지 표기

AI로 생성한 이미지를 사용한 슬라이드에는 우측 하단 텍스트 박스에 "AI로 생성된 이미지입니다" 문구가 표시된다. 실제 촬영 사진을 사용한 슬라이드에는 해당 텍스트 박스를 삭제하거나 비워둔다. 템플릿에 기본으로 포함되어 있으므로 슬라이드별로 판단해 처리한다.

인스타그램 게시

완성된 슬라이드를 이미지 파일로 내보낸 뒤 인스타그램에 게시한다.

  1. PowerPoint에서 파일 → 내보내기 → 이미지로 저장 (PNG 권장)
  2. 인스타그램 게시물 작성 시 슬라이드 이미지를 순서대로 첨부
  3. 게시물 본문 또는 댓글에 리포트 전체 링크를 추가
링크 삽입 방법

인스타그램은 게시물 본문에 링크가 클릭되지 않는다. 프로필 바이오의 링크를 업데이트하거나, Linktree 등 링크 모음 서비스를 활용해 리포트 URL을 안내한다. 게시물 본문에는 "링크는 프로필에서" 또는 "리포트 전문은 바이오 링크에서 확인" 문구를 추가한다.
